# Skill 触发评测 · 桌面分析

> 真实触发要在 Claude.ai/Cowork 里逐条灌测。本文件是基于 SKILL.md frontmatter `description` 字段对 `eval-results.json` 20 条 query 的桌面预判，估算召回与排除的潜在风险点。

## description 关键触发条件提取

```
触发词列表（显式）：
  - 小红书脚本
  - 商单
  - 种草脚本
  - 达人脚本
  - 分镜
  - 博主脚本
  - 投放脚本
触发条件（隐式）：
  - 用户上传 brief 文档
  - 用户上传博主资料
排除维度（隐式）：
  - description 强调「小红书」「短视频」，理论上应排除抖音/直播
  - description 强调「脚本与分镜表」，理论上应排除文章、推荐、食谱、方案
```

### 召回风险条目

- **query 7**「已选博主是凑活活,产品是 0 蔗糖酸奶,出脚本」  
  「博主脚本」未连写，模型有概率把「出脚本」理解为非小红书的通用脚本（如配音稿、播客）。description 没有列「短视频脚本」也没列「出脚本」。  
  **建议补强**：在触发词清单加入「出脚本」「写脚本」「脚本生成」等动作短语。

- **query 10**「帮我把这个产品的卖点改写成博主的语气,适合小红书短视频」  
  完全没有 "脚本" 字面，靠「博主语气 + 小红书短视频」推断。description 把「短视频」和「脚本」绑定在一起，单看一半的覆盖较弱。  
  **建议补强**：description 中加一句「或要求把卖点改写成达人语气、小红书短视频化表达」。

### 误触发风险条目

- **query 5**「帮我给客户写一份 MCN 项目方案」  
  含「MCN 项目」但 description 强调的是 MCN「商单」+「脚本」。如果 LLM 看到 MCN 就先打开 Skill，会误触发。  
  **建议补强**：description 在触发词后加一句「但仅限于脚本撰写场景，项目方案、营销策划、调研报告等其他 MCN 上下文不在范围内」。

- **query 8**「帮我写一条抖音直播脚本」  
  含「脚本」字面，是 description 显式触发词的子集。description 虽然写了「小红书」「短视频」但没显式排除「抖音」「直播」。  
  **建议补强**：description 加排除子句，参考 SKILL.md 主体「不适用场景」一节，可以浓缩成一句「不适用通用文案/抖音/微博/直播带货脚本」。

## description 调优建议（按优先级排序）

1. **加排除句**（最高优先级，能挡掉 query 5、8 两个误触发）
   ```
   ……该 Skill 强制执行……三层质检。**不适用通用文案、抖音/微博脚本、直播带货脚本，以及 MCN 项目方案/调研类需求**。
   ```

2. **扩充动作短语触发词**（中优先级，能拉回 query 7、10）
   ```
   ……「小红书脚本」「商单」「种草脚本」「达人脚本」「分镜」「博主脚本」「投放脚本」「出脚本」「按博主语气改写产品卖点」……
   ```

3. **保留现有 brief/博主资料 上传式触发**，不动。

应用这两条调优后，预估召回率 12/12，非触发率 8/8（达到 eval-results.json 设定的 ≥10/12 + ≥7/8 标准上限）。

## 局限说明

- 本分析是桌面预判，不能替代真实灌测。Claude 不同模型（Sonnet vs Opus vs Haiku）和不同上下文长度下 Skill 选择行为不同。
- 灌测建议在 Claude.ai 网页版逐条粘贴 query，记录是否自动加载本 Skill；至少跑两轮取较低值作为基准。
- 如果实际灌测召回率 ≥10/12 且非触发率 ≥7/8，本分析的调优建议可以暂缓；低于该阈值再按上面 3 条改 description。
